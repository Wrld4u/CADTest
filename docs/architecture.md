# Архитектура — сервис бронирования билетов

## Цель

Реализовать упрощённый API-сервис бронирования мест на концерт с защитой от race condition при высокой конкурентности запросов.

## Зафиксированный функциональный объём

- Единственный endpoint: `POST /reserve`.
- Вход: `user_id`, `seat_id`.
- Бронь постоянная (без TTL, без отмены).
- При занятом месте: `409` + сообщение `К сожалению место уже забронировано`.

## Стек

- `NestJS + TypeScript`
- `PostgreSQL + Prisma`
- `Redis` — опционально, только при необходимости оптимизации.

## Почему PostgreSQL + Prisma

Ключевое требование — строгая консистентность при одновременной записи в одно место.
Гарантия достигается за счёт уникального ограничения на `seat_id` и атомарной вставки.
Это простой и надёжный базовый подход для тестового задания.

## Структура DDD + Hexagonal

Минимальная целевая структура:

- `src/modules/reservations/domain`
  - `entities/`
  - `repositories/` (port/interface)
- `src/modules/reservations/application`
  - `dto/`
  - `use-cases/`
- `src/modules/reservations/infrastructure`
  - `http/` (controller)
  - `persistence/` (Prisma adapter)
  - `reservations.module.ts`

## Стратегия конкурентности

1. Контроллер принимает `POST /reserve`.
2. Use-case вызывает репозиторий для создания брони.
3. При наличии Redis выполняется быстрый pre-check через `SET NX` по ключу места.
4. Репозиторий выполняет атомарный `INSERT ... ON CONFLICT (seat_id) DO NOTHING RETURNING ...`.
5. Если `RETURNING` пустой, возвращается доменная ошибка занятости места.
6. Контроллер маппит эту ошибку в HTTP `409` и текст:
   `К сожалению место уже забронировано`.

## Фокус тестирования

- Unit: успешная бронь и конфликт при повторной попытке.
- E2E: конкурентные запросы к одному `seat_id` должны давать один успешный ответ и остальные `409`.

## Docker-окружение

- `docker-compose` поднимает `app + postgres`.
- Томов для Postgres нет — данные живут только в жизненном цикле контейнера.
